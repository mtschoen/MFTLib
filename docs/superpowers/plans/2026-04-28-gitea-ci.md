# Gitea CI Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire MFTLib to Gitea Actions with a Windows + Linux build/test/coverage gate, gated via branch protection on `main`.

**Architecture:** One workflow file `.gitea/workflows/test.yml` with two parallel jobs (`windows`, `linux`). Each job runs the existing local coverage script (`scripts/run-coverage.ps1 -NonInteractive` and `scripts/coverage-linux.sh` respectively). Branch protection requires both `(pull_request)` status checks. The work lands on `linux-port-mvp` (where the Linux scripts live) and is verified end-to-end via a PR to `main`.

**Tech Stack:** Gitea Actions, act_runner (host-mode Windows + Docker Linux), MSBuild, dotnet 8, cmake/ninja/g++, gcovr, coverlet, reportgenerator.

**Spec:** `docs/superpowers/specs/2026-04-28-gitea-ci-design.md`.

**Note on secrets:** This workflow does not need any secrets. Branch protection is configured via the user's admin token out-of-band. A future release/benchmark workflow may need a `secrets.CI_TOKEN` (which the parallel git-wizard work is establishing) ã out of scope here.

### Task 4: Iterate the Windows job to green

This task is unbounded ã debug whatever the runner reports until the windows job is green. Common failures and how to handle:

- [ ] **Step 1: If MSBuild fails ("MSBuild.exe : The term... is not recognized")**

Add (or restore) the `microsoft/setup-msbuild@v2` step before the coverage step.

- [ ] **Step 2: If `dotnet test` fails with TestCategory filter parsing**

The script uses `--filter "TestCategory!=RequiresAdmin"` ã pwsh tokenization is the most likely cause of breakage. If you see filter-related test discovery errors, escape the quoting inline in the YAML by switching to a `script:` style or reproduce locally first.

- [ ] **Step 3: If reportgenerator is missing**

Confirm `dotnet tool update --global dotnet-reportgenerator-globaltool` ran and that the `~\.dotnet\tools` directory is on PATH. On the Windows runner, PowerShell sessions inherit the user's PATH, but the global tools directory may need explicit export ã add:

```yaml
- name: Add dotnet tools to PATH
  shell: pwsh
  run: echo "$env:USERPROFILE\.dotnet\tools" | Out-File -FilePath $env:GITHUB_PATH -Append
```

- [ ] **Step 4: If admin tests run unexpectedly**

The script's `-NonInteractive` flag should skip them. Verify by checking the test count in the run log ã should be ~3 fewer than a local interactive run.

- [ ] **Step 5: For each fix**

Edit the workflow or script, commit with a descriptive message (`ci(windows): <what>`), push, watch the run again. Repeat until the `windows` job is green.

- [ ] **Step 6: Verify the artifact is present**

Once green, confirm the run page shows a `windows-coverage` artifact. Download it and spot-check that `coverage.xml` is non-empty (should be valid cobertura XML).
